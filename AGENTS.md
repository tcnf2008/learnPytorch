# AGENTS.md

## 项目定位

这是一个用于系统学习 PyTorch 的个人学习仓库。优先级是理解清楚、实验可复现、笔记和代码能长期积累，而不是过早工程化。

## 工作原则

- 基于第一性原理思考问题，先弄清楚概念、张量形状、数据流、梯度流和训练目标，再写代码。
- 有全局视野，大处着眼，小处着手。先定义学习目标和最小可运行实验，再逐步扩展。
- 收到任务后，先用自己的话复述：要解决什么问题、交付物是什么、有哪些不确定但已做出的假设。
- 如果存在更好的技术方案，直接说明理由，让用户决定。
- 如果任务反复报错，及时回顾最初方案是否跑偏，而不是只在局部补丁上继续叠加。

## 开发流程

- 修改前先查看当前文件、目录结构和 git 状态，避免覆盖用户已有改动。
- 保持改动范围聚焦，不顺手重构无关文件。
- 开发较大需求时，所有代码改完后再统一跑一轮测试或验证；修改过程中不要反复跑测试。
- 任务结束前，如果项目中存在 `AGENTS.md`，检查是否需要更新本文件。

## 项目结构约定

- `README.md`：学习入口、环境搭建和目录说明。
- `notebooks/`：探索式学习笔记和实验记录。
- `src/`：可复用的 Python 模块、训练循环、数据处理函数等。
- `tests/`：针对 `src/` 中可复用逻辑的测试。
- `data/raw/`：原始数据，不提交真实数据文件。
- `data/processed/`：处理中间数据，不提交真实数据文件。
- `models/`：模型权重和 checkpoint，不提交真实权重文件。
- `outputs/`：日志、图表、实验输出，不提交运行产物。

## PyTorch 学习约定

- 每个示例尽量包含：输入/输出形状、关键张量含义、损失函数、优化器和最小训练循环。
- 优先写小而完整的实验，避免只有零散代码片段。
- 涉及设备选择时，显式处理 `cpu`、`mps` 和 `cuda`，不要把本机硬件假设写死。
- 训练实验要记录随机种子、依赖版本、数据来源和关键超参数。
- 大文件、数据集、模型权重、运行日志不入库；必要时在 README 或笔记里说明如何生成或下载。

## Git 约定

- 默认只提交与当前任务相关的文件。
- 不提交虚拟环境、缓存、数据集、模型权重、实验输出和本地 IDE 配置。
- 如果需要新增会产生大量文件的工具或目录，先同步更新 `.gitignore`。
